# 前言​
很多人第一次听到 Agent 的 Tools，会自然脑补成两种极端：一种是模型真的在某个地方通过网络调用了工具，像普通程序一样 jump 进 get_weather 的实现里跑了一遍；另一种是 Tool 只是把函数说明硬塞进 Prompt，让模型模拟一个固定格式输出。两个说法都抓到了一点影子，但都不正确。​
笔者在讲 Harness 101 公开课时，发现读者最常问的不是“怎么写一个 Tool”，而是更底层的三个问题：模型到底看见了什么？它凭什么决定要调用？Harness 又在中间做了什么？ 如果这三个问题没有分清，后面讨论多轮调用、并行调用、Tool 选择、strict: true，都会像在一团雾里堆术语。我们先把最小关系画出来：​
这张图里最容易被忽略的是 Harness。模型负责在上下文里读用户问题和 Tool definitions，然后输出一个结构化的 tool_call；Tool 负责真实世界里的计算、查询、检索、读写文件；Harness 则像中间的裁判和搬运工，负责把工具说明交给模型，把模型的调用请求转成真实函数执行，再把结果塞回下一轮上下文。​
所以，Tool 不是模型突然长出了手，也不是一段普通的自然语言注释。它更像是模型与 Harness 之间的一份契约：我们告诉模型这张牌叫什么、什么时候该出、出牌时参数长什么样；模型按这套规则给出 tool_call；Harness 检查、执行、回填结果。这个过程看起来像函数调用，但本质上是一场有规则的语言游戏。​
本文就沿着这条线拆开：先把 Tool 说小，说成一份“可调用能力说明书”；再把 Tool definition 说严肃，说成一份写给模型的契约；然后看模型如何在真实 transcript 里发现、选择、调用；再回到 2023 年没有结构化 Tools API 时，开发者如何靠 Prompt 手搓这套游戏规则；最后用天气预报跑完一次调用、并行调用、多轮调用，并落到 Tool 契约该怎么调优。理解了这一层，后面再谈 Tool 调优，就不会只剩一句“模型不听话”。​
